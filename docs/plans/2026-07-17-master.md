# Vigil — Loop Engineering Hackathon Master Plan (2026-07-17)

> **For agentic workers:** Each teammate runs their own Claude agent against ONE
> of the four person plans in this folder. Use superpowers:executing-plans if the
> plugin is available; otherwise execute your plan top-to-bottom with a todo per
> task. **Never edit files outside your ownership zone (table below).**

**Goal:** Turn Vigil from a scripted frontend sim into a real autonomous
remediation loop: a mock prod service actually breaks, a real agent diagnoses it
(Zero log parsing + Akash diagnostic worker), and every destructive action goes
through a real gate (Pomerium + single-use scoped grants). Merge four branches
by 15:15, rehearse, record 3-min video, submit by 16:30.

**Judging (20% each):** idea · technical implementation · sponsor integration ·
3-min demo · agent autonomy/real-time responsiveness.

## Architecture

```
 browser ──► Next.js web (:3000)
                │  SSE /events  (LoopState snapshots)
                ▼
        vigil-agent (:4000)  ── traffic loop ──► payments-api (:4100)  [mock prod]
          │        │  reads /metrics /logs /deploys (safe, direct)
          │        │
          │        ├─► Zero.xyz        (log-parse capability, per-call, $)
          │        ├─► Akash worker (:4400 local / deployed URL)  POST /diagnose
          │        │
          │        └─► POST gate (:4200) /grants  ──► verdict + single-use token
          │
          └─ destructive calls ONLY via Pomerium (:4300) ──► payments-api
                                   /rollback /restart  (payments-api re-verifies
                                   the grant token with the gate — two layers)
```

**The layered-fallback rule (why this merges safely):** Person A implements a
local fallback for every integration, selected by env var. Unset env = fallback,
set env = real service. So after EVERY merge stage the full demo works:

| Env var | unset (fallback) | set (real) |
|---|---|---|
| `GATE_URL` | in-process policy mimic | Person B's gate service |
| `POMERIUM_URL` | direct call to payments-api | Person B's Pomerium route |
| `WORKER_URL` | in-process heuristic | Person C's Akash worker |
| `ZERO_MODE=live` | local regex parser | Person C's Zero capability |
| `OPENAI_API_KEY` | deterministic hypothesis | OpenAI-written hypothesis |

## Merge procedure (coordinator, 14:30)

Merge order matters: A is the backbone everything plugs into.

1. **A first:** `git checkout main && git merge person-a-backbone`.
   Verify: `bash` — start payments-api + vigil-agent, `curl -X POST
   localhost:4000/demo/start`, watch `curl localhost:4000/state` progress to
   `incidentStatus: "resolved"` with all fallbacks. **This alone is a working
   demo.**
2. **B:** `git merge person-b-gate`. Expected conflicts: none (new dirs only).
   Set `GATE_URL=http://localhost:4200`, `POMERIUM_URL=http://localhost:4300`.
   Re-run demo; verify the grant token flows and second use of a token fails.
3. **C:** `git merge person-c-capabilities`. Expected conflict: ONLY
   `services/vigil-agent/src/integrations/zero-live.ts` → **take C's version**
   (`git checkout --theirs`). Set `WORKER_URL`, `ZERO_MODE=live`. Re-run demo.
4. **D last:** `git merge person-d-frontend`. Set `NEXT_PUBLIC_AGENT_URL=http://localhost:4000`
   in `.env.local`, run `scripts/dev-all.sh`, open the incident page, run the
   full demo from the UI.

After each merge: `npx tsc --noEmit` must pass, demo must run. If a merge stage
breaks and can't be fixed in 15 min, ship the previous stage (fallbacks keep the
demo whole) and cherry-pick what works.

## Degradation priorities (from most to least protected)

1. Pomerium-gated rollback with a real single-use grant (the product thesis).
2. Real backbone: real 500s, real detection, real recovery curve.
3. Akash worker deployed and called live (worst case: same container local).
4. Zero live call with a receipt (worst case: fallback parser, labeled honestly).
5. LLM hypothesis step, docker-compose, extra polish.

## Honesty table (goes in README — judges respect this)

| Piece | Real | Simulated |
|---|---|---|
| payments-api errors | Real HTTP 500s served to a real traffic loop | It's a purpose-built mock service |

| Pomerium | Destructive routes really transit Pomerium; grants really single-use + TTL | — |
| Recovery | Error rate really drops because rollback really flips the service | — |

## 3-minute demo script (rehearse twice)

- 0:00 — "Vigil is an on-call engineer that holds zero production credentials."
- 0:20 — Click **Start incident**. payments-api starts throwing real 500s; the
  error-rate chart climbs from live traffic.
- 0:50 — Agent timeline: pulls deploy context, logs are unreadable → buys a
  parser call from Zero (show cost), ships evidence to the Akash worker →
  `sandbox_passed=true, recommended_action=rollback`.
- 1:40 — The gate: request hits Pomerium/gate, policy panel shows checks, grant
  is **scoped to payments-api, single-use, 60s TTL**. Rollback applies, curve
  recovers, credential shows consumed.
- 2:20 — The clamp: agent attempts mass-restart across 12 services → denied,
  blast radius over limit, policy tightened. Audit strip shows the whole story.
- 2:45 — "Every action attributed, every credential dead after one use. Zero
  standing production access."
